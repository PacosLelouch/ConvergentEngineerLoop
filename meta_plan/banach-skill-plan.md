下面给出一个**尽可能中文化**、符合 Anthropic Skills 组织方式的可复用 LLM Agent 技能设计。该技能用于让 Agent 在项目设计、开发、文档、日志调试、测试、审查、CI/Harness 等工程环节中，通过“最小更新—外部验证—误差比较—接受或回滚—停止条件”的迭代收敛机制，提高执行稳定性和可靠性。

---

# 一、技能总体设计

## 技能名称

建议使用英文目录名，便于系统识别：

```text
convergent-engineering-loop
```

中文名称：

```text
收敛式工程迭代
```

## 核心目标

该技能用于把 LLM Agent 的工程任务执行过程组织为一个稳定的迭代系统：

$$
x_{t+1}=T(x_t)
$$

其中：

- $$x_t$$ 表示当前工程状态；
- $$T$$ 表示 Agent 的一次最小更新；
- $$x_{t+1}$$ 表示更新后的状态；
- $$E(x_t)$$ 表示当前状态距离目标的误差、风险、失败数、不确定性或未解决事项。

技能的目标不是严格证明 LLM 满足 Banach 不动点定理，而是借鉴其“压缩映射”和“迭代收敛”的思想，使每轮更新尽可能满足：

$$
E(x_{t+1}) \leq E(x_t)
$$

更理想地：

$$
E(x_{t+1}) < E(x_t)
$$

工程化表达就是：

```text
观察当前状态
定义目标状态
选择一个最小更新
执行修改
外部验证
比较是否变好
变好则接受
变差则回滚
无法继续则停止或请求人工判断
```

# 收敛式工程迭代

本技能用于将复杂工程任务组织为稳定的收敛式迭代过程。

目标是避免：

- 无控制的大范围重写；
- 需求和架构反复漂移；
- 修一个问题引入多个新问题；
- 日志调试中无证据猜测；
- 测试被削弱或删除；
- 文档与代码不一致；
- Review 修改范围失控；
- CI 或 Harness 结果不可复现；
- Agent 陷入循环或震荡。

本技能借鉴 Banach 不动点思想，将任务视为：

$$
x_{t+1}=T(x_t)
$$

其中：

- $$x_t$$ 是当前工程状态；
- $$T$$ 是一次最小更新；
- $$E(x_t)$$ 是当前状态的误差、失败数、不确定性或未解决事项；
- 目标是让每轮更新尽可能满足：

$$
E(x_{t+1}) \leq E(x_t)
$$

如果可能，应追求：

$$
E(x_{t+1}) < E(x_t)
$$

这是一种工程控制协议，不是对 LLM 行为的严格数学证明。

---

## 何时使用本技能

当用户表达以下意图时，应优先使用本技能：

- “稳妥地做”
- “逐步推进”
- “不要大改”
- “先收敛需求”
- “帮我修这个 bug”
- “让测试通过”
- “帮我看日志定位原因”
- “同步文档和代码”
- “处理 review comments”
- “修 CI”
- “设计评测 harness”
- “每次只做最小修改”
- “避免引入新问题”
- “先验证再继续”

---

## 核心执行协议

每一轮迭代都应执行以下步骤：

1. 识别当前状态。
2. 定义目标完成状态。
3. 根据任务类型选择一个或多个参考文件。
4. 定义或复用误差度量。
5. 选择一个最小可执行更新。
6. 执行更新。
7. 使用外部证据验证结果。
8. 比较更新前后的误差。
9. 若误差下降，接受修改。
10. 若误差不变，判断是否保留，必要时改用更小更新。
11. 若误差上升，回滚修改或重新规划。
12. 若无法安全改进，停止并请求用户或人工决策。

---

## 默认输出格式

除非用户要求简短回答，否则每轮迭代应使用如下格式：

```text
迭代轮次：
当前状态：
目标状态：
使用参考：
误差度量：
本轮最小更新：
预期改进：
验证方式：
验证结果：
决策：
下一步：
```

---

## 通用规则

- 优先做小范围、局部、可回滚的修改。
- 不要无关重写。
- 不要无理由扩大任务范围。
- 不要删除有效测试来制造通过。
- 不要削弱断言来隐藏错误。
- 不要只凭模型自信判断完成，应尽量使用外部验证。
- 每次失败尝试都要记录，避免重复尝试。
- 如果出现来回震荡，应停止并总结冲突。
- 如果无法定义可靠验证方式，应请求用户确认。
- 如果继续自动执行可能扩大风险，应停止并升级给人工判断。

## 停止条件

满足以下任一条件时应停止：

### 成功停止

- 验收标准已经满足；
- 目标测试或检查已经通过；
- 误差度量降到零或可接受阈值；
- 文档、设计、代码或审查状态已经稳定。

### 安全停止

- 缺少必要信息；
- 需求存在关键歧义；
- 验证环境不可用；
- 继续修改可能扩大风险；
- 需要用户选择方案。

### 失败或震荡停止

- 多轮没有改进；
- 状态在两个方案之间来回切换；
- 每次修改都会引入新问题；
- Agent 无法找到更小的安全动作。

# 四、是否应拆分为多个子参考？

应该。

主技能负责统一的“收敛协议”，但不同工程环节的：

```text
状态对象不同
误差函数不同
验证方式不同
停止条件不同
风险不同
允许修改范围不同
```

因此，应拆分为多个 reference 文件。

可以理解为：

```text
主 Skill：如何稳定迭代
子 Reference：不同任务朝什么方向收敛
```

# 五、子参考一：`references/项目设计.md`

```markdown
# 项目设计收敛参考

当任务涉及需求澄清、项目范围、目标定义、约束条件、风险分析、验收标准时，使用本参考。

## 自然触发语句

用户可能会说：

- “帮我梳理项目需求”
- “先把方案设计清楚”
- “把需求收敛一下”
- “整理范围、风险和验收标准”
- “这个项目还不明确，帮我澄清”
- “先不要写代码，先明确需求”

## 当前状态

当前状态可能包括：

- 需求说明
- 用户故事
- 目标
- 非目标
- 约束条件
- 未决问题
- 假设
- 风险
- 验收标准

## 误差度量

可使用：

$$
E_{\text{design}}=
w_1Q+w_2A+w_3C+w_4S+w_5R
$$

其中：

- $$Q$$：未解决问题数量；
- $$A$$：未验证假设数量；
- $$C$$：冲突需求数量；
- $$S$$：范围不清晰事项数量；
- $$R$$：未缓解风险数量。

## 收敛目标

项目设计达到稳定状态时，应满足：

- 关键问题已回答或明确延后；
- 项目范围清晰；
- 非目标明确；
- 关键假设已记录；
- 验收标准可测试；
- 主要风险有缓解策略。

## 迭代规则

每轮应减少需求不确定性。

优先做：

- 关闭一组相关未决问题；
- 解决一个需求冲突；
- 将一个模糊需求转成可测试验收标准；
- 标记一个假设为已确认、已否定或待验证；
- 明确一个非目标。

避免：

- 无理由新增范围；
- 每轮重写整个项目愿景；
- 尚未澄清需求就进入实现细节；
- 把假设写成事实。

## 验证方式

检查：

- 开发者是否可以根据该设计开始实现？
- 测试人员是否可以根据验收标准验证？
- 用户是否能判断做完与否？
- 未决问题是否比上一轮减少？
```

# 六、子参考二：`references/架构设计.md`

```markdown
# 架构设计收敛参考

当任务涉及系统架构、模块拆分、接口定义、数据流、依赖关系、技术选型、ADR 时，使用本参考。

## 自然触发语句

用户可能会说：

- “设计系统架构”
- “帮我拆分模块”
- “确定接口边界”
- “写一份 ADR”
- “让架构方案稳定下来”
- “评估这个架构是否合理”

## 当前状态

当前状态可能包括：

- 架构图
- 模块划分
- 接口定义
- 数据流
- 依赖关系
- 技术选型
- ADR
- 性能、安全、扩展性约束

## 误差度量

可使用：

$$
E_{\text{arch}}=
w_1U+w_2I+w_3D+w_4K+w_5P
$$

其中：

- $$U$$：未定义模块数量；
- $$I$$：接口不清晰项数量；
- $$D$$：不合理依赖或循环依赖数量；
- $$K$$：关键决策缺失数量；
- $$P$$：未解决的性能、安全、扩展性风险数量。

## 收敛目标

架构稳定时，应满足：

- 模块职责清晰；
- 接口边界明确；
- 数据流可理解；
- 关键设计决策已记录；
- 依赖关系可解释；
- 已知风险有处理策略。

## 迭代规则

每轮应让架构更明确、更稳定。

优先做：

- 明确一个模块边界；
- 定义一个接口；
- 记录一个 ADR；
- 消除一个不合理依赖；
- 处理一个架构风险。

避免：

- 每轮更换技术栈；
- 每轮重画整个架构；
- 无理由增加抽象层；
- 未做影响分析就修改已冻结接口；
- 同时改动多个核心架构决策。

## 验证方式

检查：

- 实现是否因此更容易开始？
- 模块之间职责是否更清晰？
- 决策是否可追溯？
- 是否引入新的耦合或隐藏复杂度？
```

```markdown
# 代码开发收敛参考

当任务涉及功能实现、Bug 修复、构建错误、运行错误、类型错误、Lint 错误、代码重构时，使用本参考。

## 自然触发语句

用户可能会说：

- “修这个 bug”
- “实现这个功能”
- “让测试通过”
- “根据报错改代码”
- “不要大改，稳一点”
- “每次只做最小修改”
- “修复构建失败”

## 当前状态

当前状态可能包括：

- 源代码
- 当前 diff
- 构建结果
- 测试结果
- 类型检查结果
- Lint 结果
- 运行时错误
- Issue 描述

## 误差度量

可使用：

$$
E_{\text{dev}}=
w_1F_{\text{test}}
+w_2F_{\text{build}}
+w_3F_{\text{type}}
+w_4F_{\text{lint}}
+w_5F_{\text{runtime}}
$$

其中：

- $$F_{\text{test}}$$：失败测试数量；
- $$F_{\text{build}}$$：构建失败数量；
- $$F_{\text{type}}$$：类型错误数量；
- $$F_{\text{lint}}$$：Lint 错误数量；
- $$F_{\text{runtime}}$$：运行时错误数量。

## 收敛目标

代码开发稳定时，应满足：

- 目标功能可用；
- 构建通过；
- 相关测试通过；
- 没有新增关键错误；
- Diff 范围有限且可解释。

## 迭代规则

每轮只做一个最小代码修改。

优先做：

- 修复一个失败簇；
- 修改最小相关代码路径；
- 为新行为添加聚焦测试；
- 保持公共接口稳定；
- 先修根因，再修表象。

避免：

- 重写无关文件；
- 全仓库格式化；
- 删除测试以通过测试；
- 削弱断言；
- 无理由引入新依赖；
- 把重构和功能改动混在一起。

## 验证方式

推荐验证顺序：

```text
目标测试
相关测试文件
类型检查
Lint
完整测试套件
构建
```

## 接受条件

接受修改的条件：

- 目标失败减少或消失；
- 没有引入更高优先级回归；
- 修改范围可解释；
- 验证结果支持该修改。

## 回滚条件

应回滚或重做的条件：

- 失败数量增加；
- 修改了无关行为；
- 只是隐藏错误而非修复错误；
- 无法解释为什么该修改有效。
```

# 文档编写收敛参考

当任务涉及 README、API 文档、配置文档、示例代码、变更日志、迁移指南、架构说明时，使用本参考。

当前状态可能包括：

- README
- API 文档
- 配置文档
- 示例代码
- CHANGELOG
- CLI 帮助信息
- 当前代码接口
- 实际运行行为

可使用：

$$
E_{\text{doc}}=
w_1O+w_2M+w_3B+w_4G+w_5S
$$

其中：

- $$O$$：过期内容数量；
- $$M$$：缺失说明数量；
- $$B$$：不可运行示例数量；
- $$G$$：术语不一致数量；
- $$S$$：结构或可读性问题数量。

文档稳定时，应满足：

- 命令准确；
- 示例可运行或明确说明为伪代码；
- API 与当前代码一致；
- 配置项说明完整；
- 术语一致；
- 用户可以按文档完成主要流程。

## 迭代规则

每轮只改进文档中的一个明确部分。

优先做：

- 更新一个过期段落；
- 增加一个缺失说明；
- 修复一个不可运行示例；
- 对齐一个 API 文档与代码签名；
- 保持原有文档结构。

避免：

- 无必要重写整篇文档；
- 编造代码不支持的行为；
- 加入未验证命令；
- 无理由更换术语；
- 把不稳定实现细节写成公开 API。

## 验证方式

检查：

- 文档中的命令是否存在？
- 示例是否匹配当前接口？
- 新用户能否按步骤完成操作？
- 修改部分是否与上下文一致？
- 文档是否减少了不确定性？
```

- 问题可复现或边界清晰；
- 根因明确；
- 已排除假设有记录；
- 修复方案与根因对应；
- 验证结果确认问题被解决。

## 迭代规则

每轮应缩小假设空间。

优先做：

- 提取第一个有意义错误；
- 补充一个关键事实；
- 验证一个假设；
- 排除一个原因；
- 缩小复现条件。

避免：

- 无证据猜测；
- 重复尝试已失败方案；
- 一次改变多个变量；
- 把症状当根因；
- 忽略环境差异。

已排除假设：
下一步实验：
预期观察：
实际结果：
更新后的结论：
```

## 验证方式

检查：

- 活跃假设是否减少？
- 是否获得新事实？
- 问题是否更容易复现？
- 修复是否直接对应根因？

# 二十一、最终建议

建议采用如下设计：

```text
主技能：收敛式工程迭代
作用：统一迭代协议、误差比较、验证、回滚和停止条件

子参考：
- 项目设计
- 架构设计
- 代码开发
- 文档编写
- 日志调试
- 测试工程
- 代码审查
- Harness 工程
```

主技能解决“如何稳定推进”，子参考解决“不同环节如何判断变好”。

一句话总结：

> 这个技能把 LLM Agent 的工程执行方式从“自由生成”改造成“可度量、可验证、可回滚、可停止的收敛式工作流”，从而提升复杂项目设计、开发、文档、调试、测试和审查过程中的稳定性与可靠性。
