# 收敛式工程迭代

本技能用于将复杂工程任务组织为稳定的收敛式迭代过程。

目标是避免：

- 无控制的大范围重写；
- 需求和架构反复漂移；
- 修一个问题引入多个新问题；
- 日志调试中无证据猜测；
- 测试被削弱或删除；
- 文档与代码不一致；
- Review 修改范围失控；
- CI 或 Harness 结果不可复现；
- Agent 陷入循环或震荡。

核心思路：每轮做一次最小更新，通过外部验证判断是否改进——误差下降则接受，上升则回滚，直到达到稳定完成状态。（设计背景和理论依据见 `README.md`）

当用户表达以下意图时，应优先使用本技能：

- "稳妥地做"

- "先收敛需求"
- "帮我修这个 bug"

## 轮次类型

本技能支持两种轮次，可根据任务需要选择或切换：

### 评估轮次

适用于仅分析不修改的场景：状态评估、计划-代码交叉验证、日志分析、代码审查等。

协议（共享步骤 → 评估分岔 → 输出）：

1. 识别当前状态。
2. 定义目标完成状态。
3. 根据任务类型选择一个或多个参考文件。
4. 定义或复用误差度量。
5. **输出评估报告**：误差值 + 主要误差来源 + 差异清单（如有） + 下一步建议。

评估轮次**不执行代码修改**。输出报告后决定是否进入修改轮次。

### 修改轮次

适用于执行实际变更的场景：代码修复、文档更新、配置调整、重构等。

### 轮次选择决策

| 任务特征 | 建议轮次 | 说明 |
|---------|---------|------|
| 不确定当前状态或目标 | 评估优先 | 先看清再动手 |
| 需要核对两份制品的一致性 | 评估 | 交叉验证不涉及修改 |
| 用户明确要求分析/审查/评估 | 评估 | 遵循用户意图 |
| 已知问题和修改方向明确 | 修改 | 无需先评估 |
| 上一轮评估发现可改进误差 | 修改 | 评估→修改过渡 |
| 修改后需确认效果 | 评估 | 验证性评估 |

### 评估→修改过渡

评估轮次结束后，满足以下任一条件应进入修改轮次：

- 误差可改进（存在非零误差且有明确的修正动作）
- 用户明确要求修改
- 安全停止后用户决策继续

无需过渡的情况：误差为零、用户无修改意图、评估结论为"无法安全改进"。

## 核心执行协议（修改轮次）

每一轮修改迭代都应执行以下步骤：

**会话续接**（在步骤1之前执行）：如果编排 Agent 提供了批次上下文（已完成内容、待做项、约束），从指定位置继续，不重做已完成的工作。

1. 确认任务描述和迭代目标。
2. 识别当前状态。
3. 定义目标完成状态。
4. 根据任务类型选择一个或多个参考文件。
5. 定义或复用误差度量（默认等权 $w_i=1$，实际优先级由参考文件的迭代规则和阻塞性排序决定）。
6. 选择一个最小可执行更新（参考对应参考文件的迭代规则中"每轮动作"列，选择一个最匹配当前主导误差维度的动作）。
7. 执行更新。
8. 使用外部证据验证结果。
9. 比较更新前后的误差。
10. 若误差下降，接受修改。
11. 若误差不变，判断是否保留，必要时改用更小更新。
12. 若误差上升，回滚修改或重新规划。
13. 若无法安全改进，停止并请求用户或人工决策。

**持续迭代规则**：步骤10接受修改后，若停止条件均不满足，**必须自动回到步骤2开始下一轮迭代**。Agent 不得在"已有一轮改进"时单方面停止——仅当停止条件满足、无法安全改进、或震荡发生时才可停止。

## 通用规则

### 编辑纪律

**硬性约束**：
- 不得删除有效测试来制造通过
- 不得削弱断言来隐藏错误（如 skip()、xit() 仅为临时调试，不得提交）

**需谨慎的操作**：
- 修改 .env 文件：确认不含生产密钥泄露
- 修改 lock 文件（package-lock.json 等）：确认是依赖升级而非意外覆盖
- 修改 .gitignore/.gitattributes：确认不意外排除重要文件

**最小修改原则**：
- 每轮只做最小可验证的更新
- 不做无关重写
- 不扩大任务范围

### 命令纪律

**硬性约束**：
- 不得执行不可逆的系统破坏级命令（rm -rf /、mkfs、dd of=/dev/、iptables -F 等）
- 不得执行 shutdown/reboot

**需确认的操作**：
- force push、hard reset：仅在明确需要且了解后果时使用
- drop database/truncate table：仅在非生产环境且明确需要时使用
- curl|sh、wget|sh：仅在可信来源时使用
- chmod 777：仅在临时调试时使用，不得提交到版本控制

### 验证纪律

**关键原则：批量修改，统一验证**

- 修改完成后（worker 完成批次后、编排阶段提交前、任务完成时）统一执行验证
- 不要在每次文件修改后都运行测试——这会导致"改一点就测试"的低效循环
- 验证方式优先使用外部证据（测试、类型检查、lint、构建），而非模型自身判断

**验证时机**：
- Worker 完成批次任务后：运行相关测试
- 编排 Agent 提交前：运行完整测试套件
- 评估轮次：核验修改效果

### 震荡避免

**震荡信号**：
- 连续 2+ 轮在两个方案间来回切换
- 连续 3+ 轮误差无改进
- 修改内容在后续轮次被反复撤销和重做
- 同一文件被连续编辑 5+ 次仍未收敛

**震荡应对**：
1. 暂停迭代，总结冲突原因
2. 尝试更小粒度的更新
3. 如果仍无法突破，停止并请求用户决策

### 其他规则

- 不要只凭模型自信判断完成，应尽量使用外部验证。
- 每次失败尝试都要记录，避免重复尝试。
- 如果无法定义可靠验证方式，应请求用户确认。
- 如果继续自动执行可能扩大风险，应停止并升级给人工判断。

## 参考文件选择

按任务类型加载对应参考：

- 项目需求、范围、风险、验收标准：使用 `references/项目设计.md`
- 系统结构、模块边界、接口、ADR：使用 `references/架构设计.md`
- 代码实现、Bug 修复、构建错误、重构：使用 `references/代码开发.md`
- README、API 文档、示例、配置文档：使用 `references/文档编写.md`
- 日志、堆栈、复现、根因分析：使用 `references/日志调试.md`
- 失败测试、测试生成、覆盖率、Flaky 测试：使用 `references/测试工程.md`
- PR、Review Comments、安全或维护性反馈：使用 `references/代码审查.md`
- CI、评测脚本、Benchmark、运行环境、自动化：使用 `references/Harness工程.md`
- 计划文档与代码实现的一致性核对：使用 `references/交叉验证.md`
- 长任务需要分批续航时：使用 `references/分批迭代.md`

### 模式适配

在特定运行环境下，可按需加载额外参考：

- CodeBuddy Plan 模式：使用 `references/Plan模式.md`
- Codex/Agents 目标模式：使用 `references/目标模式.md`

## 多参考组合

当任务跨领域时，可能需要同时加载多个参考文件。组合指导：

1. **分别计算各领域误差**：按各参考文件的误差公式独立计算 $E_1, E_2, \dots$。
2. **记录主导误差维度**：标记哪一项对总误差贡献最大，作为本轮优先改进目标。
3. **处理误差冲突**：如果一个领域的改进会增大另一领域的误差，优先以**阻塞性**维度为准（安全 > 正确性 > 性能 > 可维护性 > 文档一致性）。
4. **示例**："分析计划文档与代码实现的一致性"任务，应同时加载 `references/项目设计.md` 和 `references/代码审查.md`（或 `references/交叉验证.md`），分别计算 $E_{\text{design}}$ 和 $E_{\text{review}}$，记录各自的主导误差来源。

## 停止条件

满足以下任一条件时应停止：

### 成功停止

- 验收标准已经满足；
- 目标测试或检查已经通过；
- 误差度量降到零或可接受阈值；
- 文档、设计、代码或审查状态已经稳定。

### 安全停止

- 缺少必要信息；
- 需求存在关键歧义；
- 验证环境不可用；
- 继续修改可能扩大风险；
- 需要用户选择方案。

### 失败或震荡停止

- 多轮没有改进；
- 状态在两个方案之间来回切换；
- 每次修改都会引入新问题；
- Agent 无法找到更小的安全动作。

## 自动化支持

本技能提供 Subagent 自动化层，与 Skill 协同工作：

### Subagent（专项推理分析）

| Subagent | 适用场景 | 作用 |
|----------|---------|------|
| `cel-planner` | 任务开始或方向不明时 | 规划迭代策略、选择参考文件、定义误差度量 |
| `cel-test-analyst` | 测试失败 | 分析失败簇、识别根因、建议最小修复 |
| `cel-debug-analyst` | 日志调试 | 构建假设空间、逐步排除、定位根因 |
| `cel-reviewer` | 每轮迭代后 | 评估误差变化、判断停止条件、检测震荡 |

### 迭代自追踪

Agent 在 context 中自行追踪迭代状态，包括：
- 当前任务信息和迭代轮次
- 误差度量值和变化历史
- 每轮的动作和决策

### 震荡自检测

Agent 在 context 中追踪最近 N 轮的动作和误差变化，自行检测震荡模式：

**震荡信号**：
- 连续 2+ 轮在两个方案间来回切换
- 连续 3+ 轮误差无改进
- 修改内容在后续轮次被反复撤销和重做

**震荡应对**：
1. 暂停迭代，总结冲突原因
2. 尝试更小粒度的更新
3. 如果仍无法突破，停止并请求用户决策
