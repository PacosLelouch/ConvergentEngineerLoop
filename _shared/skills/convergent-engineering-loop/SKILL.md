# 收敛式工程迭代

本技能用于将复杂工程任务组织为稳定的收敛式迭代过程。

目标是避免：

- 无控制的大范围重写；
- 需求和架构反复漂移；
- 修一个问题引入多个新问题；
- 日志调试中无证据猜测；
- 测试被削弱或删除；
- 文档与代码不一致；
- Review 修改范围失控；
- CI 或 Harness 结果不可复现；
- Agent 陷入循环或震荡。

核心思路：每轮做一次最小更新，通过外部验证判断是否改进——误差下降则接受，上升则回滚，直到达到稳定完成状态。（设计背景和理论依据见 `README.md`）

## 何时使用本技能

当用户表达以下意图时，应优先使用本技能：

- "稳妥地做"
- "逐步推进"
- "不要大改"
- "先收敛需求"
- "帮我修这个 bug"
- "让测试通过"
- "帮我看日志定位原因"
- "同步文档和代码"
- "处理 review comments"
- "修 CI"
- "设计评测 harness"
- "每次只做最小修改"
- "避免引入新问题"
- "先验证再继续"
- "核对计划与代码"
- "评估项目状态"

## 轮次类型

本技能支持两种轮次，可根据任务需要选择或切换：

### 评估轮次

适用于仅分析不修改的场景：状态评估、计划-代码交叉验证、日志分析、代码审查等。

协议（共享步骤 → 评估分岔 → 输出）：

1. 识别当前状态。
2. 定义目标完成状态。
3. 根据任务类型选择一个或多个参考文件。
4. 定义或复用误差度量。
5. **输出评估报告**：误差值 + 主要误差来源 + 差异清单（如有） + 下一步建议。

评估轮次**不执行代码修改**。输出报告后决定是否进入修改轮次。

### 修改轮次

适用于执行实际变更的场景：代码修复、文档更新、配置调整、重构等。

### 轮次选择决策

| 任务特征 | 建议轮次 | 说明 |
|---------|---------|------|
| 不确定当前状态或目标 | 评估优先 | 先看清再动手 |
| 需要核对两份制品的一致性 | 评估 | 交叉验证不涉及修改 |
| 用户明确要求分析/审查/评估 | 评估 | 遵循用户意图 |
| 已知问题和修改方向明确 | 修改 | 无需先评估 |
| 上一轮评估发现可改进误差 | 修改 | 评估→修改过渡 |
| 修改后需确认效果 | 评估 | 验证性评估 |

### 评估→修改过渡

评估轮次结束后，满足以下任一条件应进入修改轮次：

- 误差可改进（存在非零误差且有明确的修正动作）
- 用户明确要求修改
- 安全停止后用户决策继续

无需过渡的情况：误差为零、用户无修改意图、评估结论为"无法安全改进"。

## 核心执行协议（修改轮次）

每一轮修改迭代都应执行以下步骤：

1. 识别当前状态。
2. 定义目标完成状态。
3. 根据任务类型选择一个或多个参考文件。
4. 定义或复用误差度量（默认等权 $w_i=1$，实际优先级由参考文件的迭代规则和阻塞性排序决定）。
5. 选择一个最小可执行更新（参考对应参考文件的迭代规则中"每轮动作"列，选择一个最匹配当前主导误差维度的动作）。
6. 执行更新。
7. 使用外部证据验证结果。
8. 比较更新前后的误差。
9. 若误差下降，接受修改。
10. 若误差不变，判断是否保留，必要时改用更小更新。
11. 若误差上升，回滚修改或重新规划。
12. 若无法安全改进，停止并请求用户或人工决策。

**持续迭代规则**：步骤9接受修改后，若停止条件均不满足，**必须自动回到步骤1开始下一轮迭代**。Agent 不得在"已有一轮改进"时单方面停止——仅当停止条件满足、无法安全改进、或震荡发生时才可停止。

## 默认输出格式

### 修改轮次

除非用户要求简短回答，否则每轮修改应使用如下格式：

```
迭代轮次：
当前状态：
目标状态：
使用参考：
误差度量：
本轮最小更新：
预期改进：
验证方式：
验证结果：
决策：
下一步：
```

### 评估轮次

评估轮次使用如下格式：

```
评估类型：计划-代码交叉验证 / 状态评估 / 日志分析 / 代码审查 / ...
当前状态：
目标状态：
使用参考：
误差度量：
误差来源分析：
差异清单（如有）：
决策：是否需要进入修改轮次
下一步建议：
```

## 通用规则

- 优先做小范围、局部、可回滚的修改。
- 不要无关重写。
- 不要无理由扩大任务范围。
- 不要删除有效测试来制造通过。
- 不要削弱断言来隐藏错误。
- 不要只凭模型自信判断完成，应尽量使用外部验证。
- 每次失败尝试都要记录，避免重复尝试。
- 如果出现来回震荡，应停止并总结冲突。
- 如果无法定义可靠验证方式，应请求用户确认。
- 如果继续自动执行可能扩大风险，应停止并升级给人工判断。

- 项目需求、范围、风险、验收标准：使用 `references/项目设计.md`
- 系统结构、模块边界、接口、ADR：使用 `references/架构设计.md`
- 代码实现、Bug 修复、构建错误、重构：使用 `references/代码开发.md`
- README、API 文档、示例、配置文档：使用 `references/文档编写.md`
- 日志、堆栈、复现、根因分析：使用 `references/日志调试.md`
- 失败测试、测试生成、覆盖率、Flaky 测试：使用 `references/测试工程.md`

- CI、评测脚本、Benchmark、运行环境、自动化：使用 `references/Harness工程.md`

## 多参考组合

当任务跨领域时，可能需要同时加载多个参考文件。组合指导：

1. **分别计算各领域误差**：按各参考文件的误差公式独立计算 $E_1, E_2, \dots$。
2. **记录主导误差维度**：标记哪一项对总误差贡献最大，作为本轮优先改进目标。
3. **处理误差冲突**：如果一个领域的改进会增大另一领域的误差，优先以**阻塞性**维度为准（安全 > 正确性 > 性能 > 可维护性 > 文档一致性）。
4. **示例**："分析计划文档与代码实现的一致性"任务，应同时加载 `references/项目设计.md` 和 `references/代码审查.md`（或 `references/交叉验证.md`），分别计算 $E_{\text{design}}$ 和 $E_{\text{review}}$，记录各自的主导误差来源。

## 停止条件

满足以下任一条件时应停止：

### 成功停止

- 验收标准已经满足；
- 目标测试或检查已经通过；
- 误差度量降到零或可接受阈值；
- 文档、设计、代码或审查状态已经稳定。

### 安全停止

- 缺少必要信息；
- 需求存在关键歧义；
- 验证环境不可用；
- 继续修改可能扩大风险；
- 需要用户选择方案。

### 失败或震荡停止

- 多轮没有改进；
- 状态在两个方案之间来回切换；
- 每次修改都会引入新问题；
- Agent 无法找到更小的安全动作。

## 自动化支持

本技能提供 Hook 和 Subagent 两个自动化层，与 Skill 协同工作：

### Hook（硬约束自动执行）

| Hook | 触发时机 | 作用 |
|------|---------|------|
| `pre-edit-guard` | 文件修改前 | 检查最小修改原则、防止测试削弱、防止修改受保护文件 |
| `dangerous-command-guard` | 命令执行前 | 拦截破坏性命令（force push、rm -rf、drop database 等） |
| `post-edit-verify` | 文件修改后 | 注入验证提醒（根据文件类型建议测试、类型检查、Lint 等） |
| `oscillation-detector` | 文件修改后 | 检测震荡（连续无改进、状态来回切换），阻止循环行为 |

Hook 配置在平台设置文件中注册，Agent 无法跳过。

### Subagent（专项推理分析）

| Subagent | 适用场景 | 作用 |
|----------|---------|------|
| `convergence-planner` | 任务开始或方向不明时 | 规划迭代策略、选择参考文件、定义误差度量 |
| `test-failure-analyst` | 测试失败 | 分析失败簇、识别根因、建议最小修复 |
| `debug-root-cause-analyst` | 日志调试 | 构建假设空间、逐步排除、定位根因 |
| `review-convergence-agent` | 每轮迭代后 | 评估误差变化、判断停止条件、检测震荡 |

### 状态跟踪

迭代状态记录在 `cel-state.json` 中，包含：
- 当前任务信息和迭代轮次
- 误差度量值和变化历史
- 震荡检测结果

Hook 脚本自动读写此文件，无需手动维护。
