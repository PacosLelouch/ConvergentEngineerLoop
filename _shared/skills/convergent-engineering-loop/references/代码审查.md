# 代码审查收敛参考

当任务涉及 PR、Review Comments、安全建议、性能建议、可维护性反馈、审批前修改时，使用本参考。

## 误差度量

可使用：

$$
E_{\text{review}}=
w_1C+w_2S+w_3P+w_4M+w_5N
$$

其中：

- $C$：未解决评论数量；
- $S$：安全问题数量；
- $P$：性能问题数量；
- $M$：可维护性问题数量；
- $N$：未解释新增改动数量。

## 迭代规则

每轮处理一个评论线程或一组相关评论。

| 收敛目标 | 每轮动作 |
|---------|---------|
| 阻塞性评论已解决 | 将每条评论映射到修改或回复 |
| 非阻塞评论已处理或解释 | 如果不修改，解释原因 |
| CI 通过 | 应用最小 patch |
| Diff 保持聚焦 | 控制 PR 范围 |
| Reviewer 能容易验证修改 | 保留 Review 上下文 |

避免：

- 静默忽略评论；
- 加入无关重构；
- 未说明就改变行为；
- 没有证据就标记已解决；
- 扩大 PR 范围。

## 验证方式

检查：

- 哪些评论已经关闭？
- 是否增加新的审查负担？
- CI 是否仍然通过？
- 回复是否足够清楚？
- Reviewer 是否能快速确认？

## 交叉验证模式

当任务不是 PR 审查而是核对计划文档与代码实现的一致性时，使用 `references/交叉验证.md`。两者的区别：

| 维度 | PR 审查模式（本文件） | 交叉验证模式 |
|------|-----------|-------------|
| 主要输入 | PR diff + Review Comments | 计划文档 + 代码仓库 |
| 误差对象 | 代码质量 | 计划标记与代码实际的一致性 |
| 典型动作 | 应用 patch、回复评论 | 修正计划标记、逐项核对 |
