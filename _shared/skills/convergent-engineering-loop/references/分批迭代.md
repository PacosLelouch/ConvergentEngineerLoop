# 分批迭代收敛参考

当任务规模超出单次 Agent 会话可承载的迭代轮次时，使用本参考。

本参考由编排 Agent 加载，Worker 不需要加载。

## 是否需要分批

Agent 应根据以下因素自行判断，而非依赖硬编码阈值：
- 任务规模：todo/子目标数量
- 单项复杂度：每个 todo/子目标预估需要的迭代轮次
- 上下文消耗：每轮迭代大约消耗的上下文量
- 当前会话剩余容量

参考建议值（仅作参考，Agent 可根据实际情况调整）：

| 上下文窗口 | Plan 模式建议值 | 目标模式建议值 |
|-----------|---------------|--------------|
| 128K | todo ≥ 4 时考虑分批 | 子目标 ≥ 3 时考虑分批 |
| 200K | todo ≥ 6 时考虑分批 | 子目标 ≥ 5 时考虑分批 |
| 更大窗口 | 可适当放宽 | 可适当放宽 |

## 分批执行

分批由编排 Agent 管理。编排 Agent 维护全局状态，负责：

1. 确定批次划分（Plan 模式按 todo 依赖顺序，目标模式按子目标）
2. 向当前 Worker 传递任务上下文（目标、已完成项、约束、验收标准）
3. Worker 完成批次后，编排 Agent 收集结果并向下一个 Worker 传递续接上下文

## 批次划分

### 通用规则

- 依赖链上的 todo/子目标应在同一批次
- 可独立执行的 todo/子目标可灵活分配到不同批次

### Plan 模式批次划分

- 按依赖顺序分批，每批 3-5 个 todo
- 依赖链上的 todo 应在同一批次
- 可独立执行的 todo 可灵活分配到不同批次

### 目标模式批次划分

- 每批 2-3 个子目标
- 依赖链上的子目标应在同一批次
- 可独立执行的子目标可灵活分配

### 资源预算批次分配（目标模式）

分批执行时，资源预算应在批次间重新评估：
- 每批开始时评估剩余资源是否足以完成当前批次
- 如果资源不足，缩减当前批次范围或请求人工决策
- 每批完成后更新全局进度，为下一批重新分配预算

## Worker Spawn 提示词

编排 Agent 在 spawn 新 Worker 时，**必须**在提示词中包含以下内容，以确保 Worker 按 CEL 规范执行：

```
使用"收敛式工程迭代"技能执行以下批次任务。这是分批续接，不是全新任务——
直接从会话续接开始——跳过需求分析和任务拆分，按既定计划继续最小更新-验证-接受/回滚的迭代。

{批次交接摘要}

完成本批次所有待做项后停止，输出完成摘要（含完成项、关键决策、未解决项）。
```

关键约束：
- 提示词必须包含"收敛式工程迭代"触发词，确保 Worker 加载本技能
- 提示词必须声明"分批续接"，阻止 Worker 从头分析需求或重新拆分任务
- 批次交接摘要直接嵌入提示词，Worker 据此进入"会话续接"分支

## 批次交接

编排 Agent 在 Worker 间传递的上下文应包含：

```markdown
## 批次 N 交接摘要

### 已完成项

- [todo-1] 简述完成内容及关键决策
- [todo-2] 简述完成内容及关键决策

### 关键决策记录

- 决策1：选择了方案B而非方案A，原因...
- 决策2：在 X 模块使用了 Y 模式，原因...

### 活跃约束

- Z 接口签名不可修改（后续 todo 依赖此接口）
- 测试覆盖率需保持 ≥ 85%

### 下一批次起点

- 从 [todo-N] 开始
- 注意事项：...
```

### Plan 模式批次间进度

每批完成后，编排 Agent 在 Worker 间传递批次上下文：
- 已完成的 todo 及其验收结果
- 下一批次应从哪个 todo 开始
- 活跃约束（不可修改的接口、必须保持的测试覆盖率等）

### 目标模式批次间进度

每批完成后，编排 Agent 在 Worker 间传递批次上下文：
- 已完成的子目标及其误差值
- 当前全局误差 $E_{\text{global}}$
- 发现清单中待处理的项
- 下一批次的资源预算建议

## 批次间回滚

如果新批次发现前一批次的修改有问题：
1. 先尝试最小修复（遵循 CEL 修改轮次协议）
2. 若无法安全修复，回滚该问题修改涉及的文件
3. 在交接上下文中记录回滚原因
4. 重新评估后续批次的依赖关系

## 批次场景的 Subagent 协同

编排 Agent 在分批执行中可按需调用 CEL 分析 Subagent，调用时机为批次场景特有：

- **批次失败分析**：某批次内 Worker 修复后测试仍失败或出现新失败簇，编排 Agent 可调用 `cel-test-analyst`，将其 YAML 中的 `clusters[].suggested_fix` 交给当前 Worker 执行
- **跨批次根因排查**：批次间交接后发现根因不明（假设空间大、日志跨组件），调用 `cel-debug-analyst`，按其 `next_experiment` 指导下一批次的首个实验
- **全局收敛审查**：连续多批次无改进或批次间回滚频繁，调用 `cel-reviewer` 评估全局误差趋势，按其 `decision.recommendation` 决定是否调整批次划分或请求人工
- **大规模批次规划**：todo 数量大或跨多参考域时，调用 `cel-planner` 重新规划批次划分与误差度量

Subagent 输出的 YAML 方案中，涉及文件修改的字段由编排 Agent 交给 Worker 执行；涉及后续批次约束的字段（如 `open_questions`、`notes`）记入批次交接摘要。各 Subagent 的 YAML schema 见 `agents/instructions/cel-*.md`。

## 并发模式

多 Agent 并发执行时，各自在独立 git 分支上工作，完成后合并解决冲突。
