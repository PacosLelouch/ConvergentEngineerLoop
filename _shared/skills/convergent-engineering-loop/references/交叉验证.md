# 交叉验证收敛参考

当任务涉及计划文档与代码实现的一致性核对、设计文档与实际产出的差异分析、需求跟踪矩阵验证时，使用本参考。

## 误差度量

可使用：

$$
E_{\text{cross}}=
w_1 M + w_2 O + w_3 X + w_4 D + w_5 N
$$

其中：

- $M$：标记错误数（计划声称 done 但代码未完成或有质量问题，或计划声称 undone 但代码实际已完成）
- $O$：遗漏数（计划中要求实现但代码中完全缺失的功能点）
- $X$：过度实现数（代码中存在但计划未覆盖的额外实现）
- $D$：语义偏差数（计划与代码行为语义不一致但量级较小的差异）
- $N$：未验证项数（计划标记 done 但无测试或 CI 证据支持的项）

### 标记错误的分类

| 计划标记 | 代码实际 | 分类 | 含义 |
|----------|----------|------|------|
| done | 未实现或存在质量问题 | $M_{\text{over}}$ | 计划高估了完成度 |
| undone | 已完成 | $M_{\text{under}}$ | 计划低估了完成度 |
| undone | 确实未实现 | （不计入 $M$） | 标记正确，归属 $O$ |
| done | 确实完成且符合预期 | （不计入 $M$） | 标记正确 |
| done | 已完成但有质量缺陷 | $M_{\text{quality}}$ | 介于 done 和 undone 之间 |

## 迭代规则

每轮执行以下动作之一：

### 评估轮次（仅分析）

| 收敛目标 | 每轮动作 |
|---------|---------|
| 标记错误已修正或确认（$M = 0$） | 逐项核对计划条目，分类差异 |
| 遗漏项有明确的实现计划或已标记为非目标 | 报告 $E_{\text{cross}}$ 及各分项值 |
| 过度实现项有合理解释或已记录 | 建议下一步最小修正动作 |
| 语义偏差项已记录或修复 | — |
| 无法验证的项已标记为需要人工确认 | — |

### 修改轮次（修正差异）

优先做：

- 修正一批计划文档中的标记错误（如 undone → done）
- 为遗漏项补齐实现或标记为非目标
- 记录过度实现项的原因
- 为语义偏差添加测试或文档说明

避免：

- 评估不完整就跳过进入实现
- 仅修正标记而不验证代码实际状态
- 批量修改计划标记而不逐项核对
- 在交叉验证过程中扩大计划范围

## 交叉验证输出格式

每轮评估应逐项列出核对结果：

| 条目 | 计划标记 | 代码实际 | 判定 | 备注 |
|------|----------|----------|------|------|
| P1.1 | done | ✅ 已实现，测试通过 | 正确 | — |
| P1.2 | undone | ❌ 缺失 | 正确 undone | — |
| P2.2 | undone | ✅ 已实现 | **标记错误（应改为 done）** | 需修正计划文档 |
| P3.1 | done | ⚠️ 实现不完整 | **标记错误（应改为 partial）** | 核心逻辑缺失 |

## 验证方式

检查：

- 标记修正后的计划文档是否与代码一致？
- 遗漏项是否已排入实现计划？
- 过度实现项是否有文档记录？
- 是否存在计划标记仍然错误的新条目？
- 无法验证的条目是否已标记？

## 评估轮次决策建议

根据 $E_{\text{cross}}$ 的各分项值，建议下一步动作：

| 主导误差维度 | 建议动作 |
|-------------|---------|
| $M$ 最大 | 优先修正计划文档中的标记错误（不修改代码） |
| $O$ 最大 | 优先实现遗漏项（进入修改轮次） |
| $X$ 最大 | 审查过度实现是否合理，决定保留或回退 |
| $D$ 最大 | 逐条分析语义偏差，决定修正代码还是更新计划 |
| $N$ 最大 | 优先补齐测试或验证证据 |

## 交叉验证的 Subagent 协同

交叉验证评估轮次中，若 $E_{\text{cross}}$ 各分项难以判断是否收敛、或连续多轮无法降低标记错误数，可调用 `cel-reviewer` 做深度收敛审查：

- 将 $E_{\text{cross}}$ 各分项值（$M, O, X, D, N$）作为误差度量传入，`cel-reviewer` 输出的 `error_assessment` 反映交叉验证误差变化
- 按 `decision.recommendation` 决策：`continue` 则进入修改轮次修正主导误差维度；`success_stop` 则交叉验证通过；`oscillation_stop` 则停止并总结

`cel-reviewer` 的 YAML schema 见 `agents/instructions/cel-reviewer.md`。
