# 收敛式工程迭代——设计与背景

> 本文档包含 CEL（Convergent Engineering Loop）的设计哲学和理论背景。操作性协议见 `SKILL.md`。

## 核心思想

CEL 借鉴 Banach 不动点思想，将工程迭代视为压缩映射：

$$
x_{t+1}=T(x_t)
$$

其中：
- $x_t$ 是当前工程状态
- $T$ 是一次最小更新
- $E(x_t)$ 是当前状态的误差度量

目标是让每轮更新满足：

$$
E(x_{t+1}) \leq E(x_t)
$$

**这是一种工程控制协议，不是对 LLM 行为的严格数学证明。**

## 为什么不用严格的 Banach 定理

Banach 不动点定理要求：
1. 完备度量空间 $(X, d)$
2. 压缩映射 $T: X \to X$，存在 $q < 1$ 使 $d(Tx, Ty) \leq q \cdot d(x, y)$

工程实践中这些条件无法严格满足：
- 工程状态空间不是完备度量空间
- LLM 的"更新"不是确定性的压缩映射
- 误差函数 $E$ 难以保证连续性

CEL 保留了核心直觉——"每次更新让状态更接近目标"——但放弃了对严格收敛性的要求，转而依赖外部验证和回滚机制来保证安全性。

## 设计哲学

### 最小更新原则

每次迭代只做一个最小的、可验证的修改。理由：
- 小修改更容易验证正确性
- 小修改的副作用更可控
- 回滚小修改的成本更低
- 误差归因更清晰

### 外部验证优先

不依赖模型自身的判断来确认完成，而是使用外部证据：
- 测试通过、构建成功、Lint 清洁
- 日志证据、覆盖率数据
- CI 结果、Benchmark 数据

### 回滚安全

任何修改都应可回滚。当误差上升时，应回滚而非尝试"修补"：
- 回滚到已知良好状态
- 记录回滚原因，避免重复尝试
- 用更小的更新重试

### 震荡检测

当迭代在两个状态间来回切换时，应停止并升级：
- 检测连续 N 轮无改进
- 检测状态在两个方案间振荡
- 震荡发生时总结冲突，请求人工决策

## 三层架构

CEL 系统由三层组成：

### Skill（技能层）
- 提供操作性协议和指导
- 定义误差度量和停止条件
- 按任务类型加载参考文件

### Hook（硬约束层）
- 自动执行的不可跳过规则
- 修改前守卫：检查最小修改原则
- 危险命令拦截：阻止破坏性操作
- 修改后验证提醒：提示执行外部验证
- 震荡检测：自动检测并阻止循环行为

### Subagent（专项推理层）
- 针对特定场景的深度分析
- 收敛计划器：规划迭代策略
- 测试失败分析：分析测试失败根因
- 日志根因分析：从日志提取根因
- 收敛审查：评估迭代质量

## 多平台支持

CEL 同时支持三个 AI 编程助手平台：

| 平台 | Skills 路径 | Hooks 配置 | Agents 格式 |
|------|-----------|-----------|------------|
| CodeBuddy | `.codebuddy/skills/` | `.codebuddy/settings.json` | Markdown + YAML |
| Codex | `.agents/skills/` | `.codex/config.toml` | TOML |
| Claude Code | `.claude/skills/` | `.claude/settings.json` | Markdown + YAML |

每个平台拥有完全独立的文件副本，运行时零跨目录依赖。
